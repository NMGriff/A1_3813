This project is an Angular and Node.js chat application created for 3813ICT.

The application contains a front end built with Angular, a Node.js Express server, and a Socket.IO connection for live chat updates. The current version uses in-memory data on the server, meaning users, groups, channels and messages are reset when the Node server restarts.

## Running the project

Run the Angular front end from `3813assignment1/`:

```bash
npm install
npm run start
```

Run the Node server from `3813assignment1/server/`:

```bash
npm install
npm run start
```

### HTML and CSS layout

For Assignment 1 the focus of the application was the front end layout and basic navigation. The Angular app uses a root navigation bar in `app.component.html`. This navigation shows different buttons depending on whether the user is logged in or not.

The main chat page uses a two column layout:

| Area | Description |
| --- | --- |
| Channel sidebar | Shows the available channels, the create channel button for admins, and the list of groups. |
| Chat window | Shows the selected channel name, description, message history, loading or error messages, and the message input. |

The CSS uses grid and flex layouts. The chat page uses a grid with a fixed width sidebar and flexible chat panel. On smaller screens the layout changes to one column using a media query.

The login, register, profile, admin and create channel pages each have their own HTML template and CSS file. This keeps each page layout separate and easier to change.

### Assignment 1 data structures

At the Assignment 1 level the project can be understood as a front end chat prototype. The important data structures are users, roles, groups, channels and messages.

The current code defines these structures more fully for Assignment 2, but the Assignment 1 version is represented by the same basic fields:

| Structure | Main fields | Purpose |
| --- | --- | --- |
| User | `username`, `email`, `password`, `role`, `valid` | Represents a person who can log in. |
| Role | `super-admin`, `group-admin`, `user` | Controls which buttons and pages the user can use. |
| Group | `id`, `name`, `description`, `memberUsernames`, `channels` | Represents a user group containing members and channels. |
| Channel | `id`, `name`, `description`, `messages` | Represents a chat room inside a group. |
| Message | `id`, `author`, `time`, `body` | Represents one chat message displayed in the chat window. |

The browser stores the currently logged in user in local storage under `currentUser`. This allows the page to remember the login state until logout is clicked.

## Assignment 2

### Main data structures

Assignment 2 expands the application into a working Angular and Node.js chat system. The main server data is in `3813assignment1/server/server.js`.
The server stores the main objects in arrays, while Angular uses TypeScript types to describe the data returned from the API.

### Users

Users are represented by the `User` class on the server.

```ts
{
  username: string,
  birthdate: string,
  age: number,
  email: string,
  password: string,
  role: 'super-admin' | 'group-admin' | 'user',
  valid: boolean
}
```

The server starts with three users:

| Username | Email | Role |
| --- | --- | --- |
| Alan | `admin1@gmail.com` | `super-admin` |
| Phil | `admin2@gmail.com` | `group-admin` |
| Frank | `admin3@gmail.com` | `user` |

The `publicUser` helper removes the password before user data is returned to the Angular front end.

### Groups

Groups are stored in the `groups` array. A group contains basic group information, a list of member usernames, a list of pending usernames, and a list of channels.

```ts
{
  id: string,
  name: string,
  description: string,
  createdBy: string,
  memberUsernames: string[],
  pendingUsernames: string[],
  isMember: boolean,
  isPending: boolean,
  canManage: boolean,
  channels: ChatChannel[]
}
```

The default group is `global`, with Alan, Phil and Frank as members.

The `isMember`, `isPending` and `canManage` values are calculated when groups are returned to Angular. They are used so the front end knows which buttons to show for the current user.

### Channels

Channels are stored inside groups.

```ts
{
  id: string,
  groupId: string,
  groupName: string,
  name: string,
  description: string,
  bannedUsernames: string[],
  messages: ChatMessage[]
}
```

A channel belongs to a group. Users must be group members and must not be banned from the channel before they can send messages.

The server stores channels inside groups. When channels are returned to Angular, the server also includes `groupId` and `groupName` so the chat page can show channels as `Group / Channel`.

### Messages

Messages are stored inside a channel's `messages` array.

```ts
{
  id: number,
  author: string,
  time: string,
  body: string,
  createdAt: string,
  isCurrentUser?: boolean
}
```

The server uses `nextMessageId` to give each new message a unique id. The optional `isCurrentUser` field is added by Angular for display, so the interface can tell whether the message was written by the logged in user.

### REST API

The Angular front end communicates with the Node.js server at `http://localhost:3000/api`.
The API routes are implemented in `3813assignment1/server/server.js` and are called from `AuthService` and `ChatService`.

Common error responses look like this:

```ts
{
  valid: false,
  error: string
}
```

| Method | Route | Parameters | Return value | What it does |
| --- | --- | --- | --- | --- |
| `POST` | `/api/auth` | Body: `email`, `password` | Valid user object or `{ valid: false }` | Logs a user in by matching email and password. |
| `POST` | `/api/register` | Body: `username`, `birthdate`, `age`, `email`, `password` | Valid user object or `{ valid: false }` | Creates a new user if the username and email are not already used. |
| `GET` | `/api/users` | Query: `username` | `ChatUser[]` | Returns all public users. Requires a valid viewer username. |
| `POST` | `/api/users/:username/role` | URL: target `username`. Body: `actingUsername`, `role` | Updated public user | Lets a Super Admin change a user's role. |
| `DELETE` | `/api/users/:username` | URL: target `username`. Query or body: `actingUsername` | `{ valid: true }` | Deletes a user. Super Admins can delete other users. A user can delete themself. |
| `GET` | `/api/groups` | Query: `username` | `ChatGroup[]` | Returns groups with membership, pending and management flags for the viewer. |
| `POST` | `/api/groups` | Body: `actingUsername`, `name`, `description` | Created group | Creates a group. Only Group Admins and Super Admins can create groups. |
| `DELETE` | `/api/groups/:groupId` | URL: `groupId`. Query or body: `actingUsername` | `{ valid: true }` | Deletes a group if the acting user can manage it. |
| `POST` | `/api/groups/:groupId/interest` | URL: `groupId`. Body: `username` | Updated group | Adds the user to the group's pending request list. |
| `POST` | `/api/groups/:groupId/members` | URL: `groupId`. Body: `actingUsername`, `username` | Updated group | Adds a user to a group and removes them from the pending list. |
| `DELETE` | `/api/groups/:groupId/members/:username` | URL: `groupId`, target `username`. Query or body: `actingUsername` | Updated group | Removes a user from a group. Users can remove themself, and admins can remove group members. |
| `POST` | `/api/groups/:groupId/channels` | URL: `groupId`. Body: `actingUsername`, `name`, `description` | Created channel | Creates a channel inside a group. |
| `DELETE` | `/api/groups/:groupId/channels/:channelId` | URL: `groupId`, `channelId`. Query or body: `actingUsername` | `{ valid: true }` | Deletes a channel from a group. |
| `POST` | `/api/groups/:groupId/channels/:channelId/ban` | URL: `groupId`, `channelId`. Body: `actingUsername`, `username` | Updated channel | Bans a group member from a channel and emits an admin report event. |
| `GET` | `/api/channels` | Query: `username` | `ChatChannel[]` | Returns all channels the user can access. |
| `POST` | `/api/channels` | Body: `actingUsername`, `groupId`, `name`, `description` | Created channel | Creates a channel. This is the route used by `ChatService.createChannel`. |
| `GET` | `/api/channels/:channelId/messages` | URL: `channelId` | `ChatMessage[]` | Returns messages for the first channel with that id. This is a more general route. |
| `GET` | `/api/groups/:groupId/channels/:channelId/messages` | URL: `groupId`, `channelId` | `ChatMessage[]` | Returns messages for one channel inside one group. |
| `POST` | `/api/channels/:channelId/messages` | URL: `channelId`. Body: `author`, `body` | Created message | Sends a message to the first channel with that id if the author is allowed. |
| `POST` | `/api/groups/:groupId/channels/:channelId/messages` | URL: `groupId`, `channelId`. Body: `author`, `body` | Created message | Sends a message to a specific group channel. This is the route used by `ChatService.sendMessage`. |

### Socket.IO events

Socket.IO is used in addition to the REST API so the page can update without being manually refreshed.

| Event | Sent by server when | Used by front end for |
| --- | --- | --- |
| `message:created` | A message is created. | Adding the new message to the matching channel. |
| `channel:created` | A channel is created. | Adding the new channel to the channel list. |
| `groups:changed` | Groups, members or channels change. | Reloading groups and channels. |
| `users:changed` | User roles or users change. | Reloading the user list in the admin page. |
| `admin:report` | A user is banned from a channel. | The event is emitted by the server, but there is no current Angular listener for it. |

### Angular architecture

The Angular application is built with standalone components. Routing is defined in `3813assignment1/src/app/app.routes.ts`, and pages import the Angular modules and child components they need.

| `LoginComponent` | Login form. Calls `AuthService.login`, saves the returned user and navigates to the profile page. |
| `RegisterComponent` | Registration form. Checks password confirmation, calls `AuthService.register`, saves the returned user and navigates to the profile page. |

| `AdminComponent` | Permissions page for Group Admins and Super Admins. Handles users, roles, groups, members, channels and channel bans. |
| `CreateChannelComponent` | Separate channel creation page. Loads groups the current user can manage and creates a channel in the selected group. |

| --- | --- |
| `MessageComponent` | Displays one chat message. |
| `MessageHistoryComponent` | Displays a list of messages and scrolls to the bottom when messages change. |

### Services

| Service | Purpose |
| --- | --- |
| `AuthService` | Handles login, registration, local storage, logout, role checks and logged in checks. |

| Guard | Purpose |
| --- | --- |
| `authGuard` | Checks `AuthService.isLoggedIn()`. If the user is not logged in, it redirects them to `/login`. |

### Component and service flow

The page components do not directly make API requests. Instead, they call `AuthService` or `ChatService`. This keeps API logic in the services and keeps the components focused on page state and user interaction.

For example, when a user sends a message:

1. `MessageInputComponent` emits the typed message.
2. `HomeComponent` receives the event and gets the current user from `AuthService`.
3. `HomeComponent` calls `ChatService.sendMessage`.
4. `ChatService` sends a `POST` request to the Node server.
5. The server stores the message in the channel and emits `message:created`.
6. `HomeComponent` receives the Socket.IO event and adds the new message to the displayed channel.

This same pattern is used throughout the app. Components handle screens and forms, services handle data access, and the server holds the main application data.
